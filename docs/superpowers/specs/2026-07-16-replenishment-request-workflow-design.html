OMS ↔ WMS Architecture

보충 요청 승인 워크플로우

OMS는 보충을 요청하고, WMS는 창고 작업을 승인·실행한다. 재고 원장과 처리 이력의 책임을 WMS로 되돌리는 설계.

배경과 목표

현재 OMS 관리자 화면은 WMS의 창고 작업을 직접 원격 변경한다.

  • wmsInventoryAdapter.adjust(...) → WMS POST /api/inventory/adjust
  • wmsPurchaseOrderAdapter.create(...) → WMS POST /api/purchase-orders
  • wmsPurchaseOrderAdapter.receive(...) → WMS POST /api/purchase-orders/receive

OMS가 다른 바운디드 컨텍스트인 WMS의 실물 수량·발주·입고를 직접 손대면서 창고 권한이 요청자에게 새어 있다.

목표 OMS는 보충을 요청하고 WMS 관리자가 검토 후 승인 또는 반려한다. 요청과 처리 이력의 원본은 WMS가 보관하고 OMS는 API로 조회한다.

데이터 모델 — WMS 원본

ReplenishmentRequest

필드설명
idPK, 내부 요청번호
requestKeyOMS 요청 폼을 열 때 한 번 생성한 UUID, UNIQUE 멱등 키
reason요청 사유, 필수
statusREQUESTED / APPROVED / REJECTED / FULFILLED
requestedAt요청 접수 시각
decidedAt승인 또는 반려 시각
fulfilledAt연결된 발주의 입고 완료 시각
wmsMemo승인 시 선택, 반려 시 필수인 WMS 처리 메모
purchaseOrderId승인 시 생성된 발주 연결, nullable + UNIQUE

processedAt 하나로 합치지 않는다. 승인 시각과 입고 완료 시각은 서로 다른 사건이며 그 사이에는 실제 리드타임이 있다.

ReplenishmentRequestItem

필드설명
productId필수이며 WMS 재고에 존재해야 하는 품목
requestedQty1 이상의 요청 수량

다품목 요청은 기존 다라인 PO로 1:1 매핑한다. 한 요청 안에서 같은 productId를 여러 라인으로 보내는 것은 거부한다.

멱등성과 접수 검증

OMS 요청 전송
→ WMS 저장 성공
→ 응답 유실
→ OMS 재전송
→ requestKey가 없다면 서로 다른 PK의 중복 요청 2건

requestKey는 폼 GET에서 한 번 생성해 hidden 필드로 전달한다. 컨트롤러와 어댑터는 새 키를 만들지 않으며 전송 내부 재시도에서도 같은 키를 사용한다.

  • 동일 키 + 동일 내용 → 기존 요청 반환
  • 동일 키 + 다른 내용 → 409 Conflict
  • 신규 키 → REQUESTED 저장 후 반환

내용은 중복 품목을 거부한 뒤 품목 순서와 무관한 Map<productId, requestedQty>와 reason으로 비교한다. UNIQUE 제약이 최종 방어선이며 동시 삽입 경쟁도 기존 행을 다시 조회해 수렴시킨다.

WMS 접수 경계 검증

  • requestKey와 공백이 아닌 reason 필수
  • 품목 한 개 이상
  • 필수 productId가 WMS 재고에 존재
  • requestedQty 1 이상
  • 동일 productId 중복 라인 거부

상태 전이

REQUESTED접수
APPROVEDPO 생성
FULFILLED입고 완료
REJECTED반려
REQUESTED ──승인 / PO 생성──> APPROVED ──PO 입고──> FULFILLED
     └────────── 반려 ──────────> REJECTED
  • 승인·반려는 REQUESTED에서만 허용한다.
  • 도메인 서비스가 잘못된 재처리를 거부하고 관리자 컨트롤러는 오류 flash 후 목록으로 redirect한다.
  • 반려에는 공백이 아닌 wmsMemo가 필수이며 승인 메모는 선택이다.
  • 승인은 ORDERED PO 생성까지만 수행하며 입고와 재고 증가는 실제 물건 도착 시 처리한다.
  • PO 입고 트랜잭션에서 연결 요청이 있을 때만 FULFILLED로 전이한다. 직접 생성 PO도 정상 입고된다.
  • OMS 재입고 콜백은 기존처럼 커밋 후 실행한다.

전체 흐름

OMS 관리자
  재고 조회 화면 진입 → requestKey UUID 1회 생성(hidden)
  → 품목·수량·사유 입력 → POST /api/replenishment-requests
                           → WMS: REQUESTED 저장

WMS 관리자
  검토 → 반려: REJECTED + wmsMemo + decidedAt
       → 승인: PurchaseOrderService.create → ORDERED PO
               APPROVED + purchaseOrderId + decidedAt

WMS 관리자
  실제 물건 도착 → 기존 발주 화면에서 receive
  → 한 트랜잭션: 재고 증가 + 연결 요청 FULFILLED + fulfilledAt
  → 커밋 후: OmsReplenishmentNotifier → OMS 백오더 승격

OMS 관리자
  GET /api/replenishment-requests → 요청 상태와 처리 이력 확인

API와 화면 액션

WMS 신규

MethodURL인증 / CSRF용도
POST/api/replenishment-requestsBasic, CSRF 예외멱등 요청 접수
GET/api/replenishment-requestsBasicOMS 이력 조회
POST/admin/replenishment-requests/{id}/approveBasic + CSRF승인 후 PO 생성
POST/admin/replenishment-requests/{id}/rejectBasic + CSRF반려

관리자 액션 승인·반려 POST는 REST가 아닌 HTML 폼 액션이다. 성공과 도메인 오류 모두 목록 화면으로 redirect하고 오류는 flash 메시지로 표시한다.

OMS 신규

  • 폼 GET에서 requestKey를 생성해 hidden 필드로 제공
  • 폼에서 받은 키를 그대로 전달하는 보충 요청 어댑터
  • 요청 이력 조회 어댑터
  • 기존 재고 조회를 유지한 보충 요청 폼·이력 화면

삭제 대상

OMS

  • InventoryAdminController의 adjust, PO create/receive 액션과 PO 관리자 화면
  • WmsInventoryAdapter.adjust() 메서드만 제거 — reserve/ship/release 때문에 클래스는 유지
  • WmsPurchaseOrderAdapter 전체와 관련 폼·DTO·템플릿

WMS

  • InventoryController의 POST /api/inventory/adjust만 제거
  • PurchaseOrderController 전체와 전용 Request/Response DTO, 컨트롤러 테스트 제거
  • PurchaseOrderService, 폼, 도메인, 저장소는 WMS admin과 승인 흐름 때문에 유지

삭제 직전 각 대상의 참조를 다시 검색해 호출자 0을 확인한다.

네이밍

  • OMS→WMS: 보충 요청 — ReplenishmentRequest
  • WMS→OMS: 재입고 통지(callback) — 기존 OmsReplenishmentNotifier

기존 클래스명 변경은 하지 않는다. 코드 주석과 문서의 용어 구분으로 충분하다.

테스트

WMS

  • REQUESTED→APPROVED→FULFILLED, REQUESTED→REJECTED와 잘못된 전이 방어
  • 승인 시 ORDERED PO 생성과 purchaseOrderId 연결
  • 입고 시 연결 요청만 FULFILLED, 직접 생성 PO도 정상 입고
  • 같은 requestKey·같은 내용은 기존 요청 반환, 다른 내용은 409
  • 빈 reason/items, 잘못된 productId·수량, 중복 productId 거부
  • 무자격 401, 무CSRF 403, 관리자 성공·오류 3xx redirect와 flash

OMS

  • 폼 GET에서 키 생성 후 POST와 재시도에서 같은 키 유지
  • Basic Auth를 포함한 요청·이력 조회 어댑터
  • 폼 입력 검증과 WMS 오류 표시

배포 순서

  1. WMS 확장 배포
    신규 요청 API·도메인·검토 화면을 추가하되 기존 관리자 REST는 임시 유지한다.
  2. OMS 전환 배포
    수동 조정·발주·입고 화면과 어댑터를 제거하고 보충 요청 화면으로 교체한다.
  3. WMS 정리 배포
    OMS 전환을 확인한 뒤 기존 관리자 REST와 전용 DTO·테스트를 제거한다.

코드 작업 범위에는 기존 API 삭제까지 포함하지만 WMS 확장과 정리는 서로 다른 배포로 내보낸다. 신규 테이블은 현재 운영 정책대로 ddl-auto: update로 생성한다.
